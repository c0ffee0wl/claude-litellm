# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

claude-litellm is a Linux-side setup repo that installs and configures the [LiteLLM Proxy](https://docs.litellm.ai/) AI gateway as a systemd user service on `127.0.0.1:4000`, plus optionally Claude Code with managed settings. LiteLLM speaks Anthropic's Messages format natively on `/v1/messages` and translates to Azure OpenAI / Vertex Gemini / etc. via `model_list` entries.

There is no build system, test suite, or linter — this is a deployment automation repo of Bash scripts plus YAML/JSON configs.

## Architecture

Single execution context: Linux. Runs as the current `$USER` (no devuser, no sudo for the script itself; `sudo` is invoked only for `apt`, `loginctl enable-linger`, and writing `/etc/claude-code/managed-settings.json`).

Entry point: `linux/setup.sh`. It sources `linux/common.sh` for utilities and runs ten phases. Key flags (`--router-only`/`--harden-only` mutually exclusive; `--install-obsidian` and `--docker` additive — except `--docker` + `--harden-only`, rejected since harden-only installs no LiteLLM; not persisted across reruns):
- `--router-only` skips Phase 8a (system-level managed-settings.json sudo-install) and Phase 8e (nah Claude Code plugin — also a policy layer); Phase 8b (`/tmp/claude` sandbox prereq, no sudo) now runs in **all** modes so the router-only `/sandbox` opt-in isn't missing its prereq dir. User-level Phase 8c–8d (`~/.claude/settings.json` statusLine, `~/.claude/statusline.sh`) still run, but Phase 8c keeps the full `sandbox` block and sets `enabled:false` (`jq '.sandbox.enabled = false'`, **not** stripped) so router-only ships the sandbox **off** by default with the credential/network floor pre-configured — flip `enabled:true` to turn it on. The bwrap runtime is installed in all modes (Phase 2). Router-only also skips `CLAUDE_CODE_SUBPROCESS_ENV_SCRUB` (Phase 5a, removed if present) so a dev box keeps the full subprocess env (incl. the gateway token) for Bash/MCP/hooks. LiteLLM + Claude Code + claude-devtools all install locally — useful for a dev box where you want to route through LiteLLM without system-level policy enforcement.
- `--harden-only` is the inverse: skips Phases 4a (LiteLLM), Phase 5b (provider-secret collection), Phase 6 (Postgres + LiteLLM env file), Phase 7 (`litellm.service`), and Phase 9 (`claude-devtools.service`). Phase 5a's `~/.profile` writes still run; Phase 10's cleanup-only uninstaller for the legacy claude-run / claude-history service runs in every mode. Installs only Claude Code + managed-settings — useful when LiteLLM runs on another host. To point at a remote router, set `ANTHROPIC_GATEWAY_URL=http://<remote>:4000` in `.env` before running — `set -a` sourcing lets `.env` override the localhost default, which then lands in `~/.profile` as `ANTHROPIC_BASE_URL`.
- `--install-obsidian` is additive (combinable with any mode, not part of the mutual-exclusivity check): it enables Phase 4c (ACP adapter via `bun add -g`) and Phase 4d (latest Obsidian amd64 `.deb`). ACP is **only** installed with this flag — it is no longer a full-mode default. The ACP bridge exists to drive Claude Code from editors like Obsidian, so the two are paired behind one flag.
- `--docker` is additive (combinable with any mode except `--harden-only`): it runs **LiteLLM** as a rootless Docker Compose service instead of the native uv install. Enables Phase 2b (`install_docker_rootless` in `common.sh`), skips Phase 4a (uv litellm + `prisma generate` — the image bundles both), points Phase 6's `DATABASE_URL` at the **host** Postgres' Unix socket (`?host=/run/postgresql`, which stays native), and makes Phase 7 deploy `litellm-docker-compose.yml` + the `litellm-docker.service` unit (`docker compose up`) under the canonical `litellm` user-service name. **Only LiteLLM is containerized** — Postgres stays on the host (so `/ui` models/keys/spend persist across the native↔docker switch — no fresh DB volume) and **claude-devtools stays native** (Phase 9; its vite build OOMs the same in `docker build`, and the image isn't built on-host anyway). Rootless is deliberate (no root daemon, no `docker` group ≈ root) — see *Docker mode (`--docker`)* below.

API keys come from `.env` in the repo root (gitignored; create from `.env.example`) — or from any provider env var you happen to have exported in your shell / `~/.profile`. `.env` is optional: if it's missing, Phase 5 logs a warning and continues, picking up whatever is already exported. `setup.sh` Phase 5 auto-discovers them via `collect_litellm_provider_vars` in `common.sh` (pattern `*_API_{KEY,BASE,VERSION,TOKEN}` plus a named list for AWS/GCP/watsonx/Azure-AD/OpenAI-org/HF extras) and writes the matches plus the LiteLLM master key + DB URL **only** to `~/.config/litellm/env` (the systemd EnvironmentFile, mode 600). The user's `~/.profile` gets only the gateway URL, the master key as `ANTHROPIC_AUTH_TOKEN`, and telemetry opt-outs — no upstream provider secrets.

LiteLLM speaks Anthropic on `/v1/messages` (the unified endpoint, not the `/anthropic/*` pass-through). Claude Code is configured via `ANTHROPIC_BASE_URL=http://127.0.0.1:4000` (no path suffix).

|---|---|
| 0 | self-update: `git pull --ff-only` and re-exec if behind. Skipped if `claude`/ACP is running or not a git checkout. |
| 1 | shell profile setup: ensure `~/.profile` and the `~/.bash_profile` shim sourcing it exist |
| 2 | apt: `git curl jq ca-certificates unzip rsync` + `bubblewrap socat ripgrep` (all modes, + `configure_bwrap_apparmor`). The sandbox runtime ships everywhere so a user can opt into `/sandbox` via the UI even under `--router-only` (which ships the sandbox **off** by default — Phase 8c sets `enabled:false` but keeps the block, and the runtime stays installed). The packages are tiny + benign. `ripgrep` is an undocumented `/sandbox` dependency — the Mode/Overrides toggle tabs only render once it resolves (anthropics/claude-code#31804, #31708); a real `rg` binary keeps the sandbox toggle reachable |
| 2b | (only with `--docker`) `install_docker_rootless` (common.sh): install Docker CE from the official apt repo (GPG key + `download.docker.com` sources, Kali→trixie fallback — apt-repo block reused near-verbatim from `/opt/linux-setup`), then set it up **rootless** (`dockerd-rootless-setuptool.sh install` → `systemd --user` `docker.service`; **no** `docker` group). If a rootful daemon is already running (writable `/var/run/docker.sock`), the setuptool is invoked with `--force` so rootless coexists alongside it on a separate per-user socket (forced only when that socket is present, so a clean box keeps the setuptool's RootlessKit smoke-test fatal). Installs `docker-ce{,-cli} containerd.io docker-{buildx,compose}-plugin docker-ce-rootless-extras uidmap slirp4netns dbus-user-session`. Idempotent (skips the **engine** install if `docker` present but still ensures the rootless + compose deps `uidmap slirp4netns dbus-user-session docker-ce-rootless-extras docker-compose-plugin` via a `dpkg -s` check — a pre-existing engine commonly lacks `uidmap` (setuptool aborts "Missing system requirements … apt-get install -y uidmap") or, on a `docker.io` host, the `docker compose` plugin the litellm unit's `docker compose up`/`down` need; skips setuptool if `~/.config/systemd/user/docker.service` exists) |
| 3 | Install bun + uv (install-if-missing-only); symlink `node→bun`, `npx→bunx` |
| 4a | LiteLLM via `uv tool install litellm` (`--with 'litellm[proxy,proxy-runtime]>=1.84.0'`) → `~/.local/bin/litellm` (install-if-missing). **Skipped under `--docker`** (the `ghcr.io/berriai/litellm` image bundles LiteLLM + a generated Prisma client). The `>=1.84.0` **floor** (not a hard pin) sits above the compromised 1.82.7/1.82.8 PyPI releases (so the old explicit `!=` excludes are gone) and guarantees the GPT-5.4 reasoning fixes this setup relies on: the chat→Responses auto-route (1.83.0+) and `output_config.effort`→`reasoning_effort` mapping (1.83.1+) |
| 4b | Claude Code via official curl-pipe installer (install-if-missing); always runs |
| 4c | (only with `--install-obsidian`) ACP adapter via `bun add -g` (install-if-missing) |
| 4d | (only with `--install-obsidian`) latest Obsidian amd64 `.deb` resolved from the GitHub releases API via `jq`, downloaded with `curl_secure` and installed via `sudo apt-get install` (apt resolves deps). install-if-missing (skips if `dpkg-query` reports `obsidian` as `install ok installed`); failures non-fatal |
| 5a | gateway URL + master key (as `ANTHROPIC_AUTH_TOKEN` and `LITELLM_API_KEY`) + telemetry opt-outs + `CLAUDE_CODE_SUBPROCESS_ENV_SCRUB` + `CLAUDE_CODE_ENABLE_GATEWAY_MODEL_DISCOVERY` + `ANTHROPIC_DEFAULT_{HAIKU,SONNET,OPUS}_MODEL` + their `_SUPPORTED_CAPABILITIES` companions → `~/.profile` (kept here, not managed-settings, so user can `unset` for `--dangerously-skip-permissions` and so `--router-only` gets them without root). The three default-model vars resolve to `azure/gpt-5.4{,-mini}` when both `AZURE_OPENAI_API_KEY` and `AZURE_RESOURCE_ENDPOINT` are set in `.env`; otherwise they're written empty (any prior non-empty value in `~/.profile` is preserved across re-runs) and a yellow end-of-script banner instructs the user to add a model via `/ui` and fill them in. The `_SUPPORTED_CAPABILITIES` vars (`thinking,adaptive_thinking,interleaved_thinking,effort,xhigh_effort`; haiku omits `interleaved_thinking` + `xhigh_effort`) tell Claude Code to enable extended thinking + effort for the non-`claude-*` ids — without them detection stays off and no thinking block is ever sent. `xhigh_effort` is safe at the LiteLLM `>=1.84.0` floor: GPT-5.4 accepts `reasoning_effort=xhigh` (Azure v1 API ref) and LiteLLM clamps levels the model map doesn't declare (max→xhigh→high, BerriAI/litellm#26111), so while the `azure/` map flags lag, xhigh degrades to high — never a 400 (the map auto-refreshes at boot, so it activates upstream-side without a re-run). `max_effort` is intentionally excluded: `max` is Anthropic-only and would always clamp — a permanently misleading picker entry. They track the model vars (written empty when Azure is blank, manual values preserved). Also scrubs `IS_DEMO` from `~/.profile` + current shell (anthropics/claude-code#37780). Under `--router-only`, `CLAUDE_CODE_SUBPROCESS_ENV_SCRUB` is **not** set (removed if present) so a dev box's Bash/MCP/hooks subprocesses keep the full env (incl. the gateway token) |
| 5b | provider secrets from `.env`/shell/`~/.profile` + master key → in-memory `LITELLM_ENV_CONTENT` (written to disk in Phase 6). Uses `collect_litellm_provider_vars` |
| 6 | Postgres + LiteLLM env file: apt-install `postgresql`; create role + db idempotently (skipped if `DATABASE_URL` set externally in `.env`); resolve `LITELLM_DB_PASSWORD` (reused across reruns, else `openssl rand -hex 24`); append `DATABASE_URL` / `STORE_MODEL_IN_DB=True` / `LITELLM_DB_PASSWORD` to the in-memory env content from 5b and write `~/.config/litellm/env` atomically (mode 600). Enables GUI model management via `/ui`. Postgres stays on the host **in all modes incl. `--docker`** — only the transport differs: native uses TCP loopback (`@127.0.0.1:5432`), `--docker` uses the host Postgres **Unix socket** (`@localhost/litellm?host=/run/postgresql`, bind-mounted into the container — no TCP/network exposure) and calls `ensure_pg_socket_scram_rule` (common.sh) to add a scoped `local litellm litellm scram-sha-256` pg_hba line since Debian defaults `local` to peer auth |
| 7 | Deploy `litellm-config.yaml` to `~/.config/litellm/`; install + start `litellm.service`. LiteLLM auto-runs Prisma migrations on first boot. **Under `--docker`** instead deploys `litellm-docker-compose.yml` → `~/.config/litellm/docker-compose.yml` + the `litellm-docker.service` unit (`docker compose up`/`down`, `DOCKER_HOST` = rootless socket) under the same `litellm` user-service name; `wait_for_litellm` + UI/port stay identical (loopback-published `127.0.0.1:4000`) |
| 8a | (skipped under `--router-only`) Token-substitute managed-settings → `/etc/claude-code/managed-settings.json` (sudo). Managed-settings carries **no** `sandbox` block — only `permissions` + `hooks` (see *Sandbox control* below: the sandbox lives in user settings so `/sandbox` can toggle it) |
| 8b | (all modes, no sudo) create `/tmp/claude` sandbox prereq dir. Runs everywhere now that the sandbox runtime ships in all modes — incl. `--router-only`, where the user can `/sandbox` on |
| 8c–8d | User-level Claude Code state (runs in every mode): `~/.claude/settings.json` (statusLine + the full `sandbox` block — deployed fresh-install only, never clobbered on re-runs; `enabled:true` in full/harden, `enabled:false` under `--router-only` (`jq '.sandbox.enabled = false'`, block kept **not** stripped → floor pre-configured, flip to enable). On an **existing** settings.json that lacks a sandbox block, setup never rewrites the file — the end-of-setup banner points at `linux/scripts/add-sandbox-block.sh`, an idempotent (`//=`) add-if-missing merge the user runs explicitly; `--disabled` under router-only), `~/.claude/statusline.sh` |
| 8e | (skipped under `--router-only`) Idempotent `claude plugin marketplace add manuelschipper/nah && claude plugin install nah@nah --scope user`. Deterministic action-aware PreToolUse guard from [manuelschipper/nah](https://github.com/manuelschipper/nah). Classifies commands into action types (`filesystem_delete`, `lang_exec`, `git_history_rewrite`, …) and resolves `allow`/`ask`/`block` with sensitive-path + content-scan context. Closes wrapper-evasion gaps the `Bash(...)` regex denies miss (`bash -c "rm -rf …"`, `python -c`, `git push -f` short form, `xargs rm`, `find -delete`). Install enables it by default + the absent branch re-asserts enable; a plugin found **disabled** is treated as a deliberate user opt-out and left disabled (no re-enable). Unpinned ref — `claude plugin update nah --scope user` to upgrade. |
| 9 | Skipped if `MemTotal < 3.5 GB` (vite's renderer build OOMs on smaller boxes; the end-of-setup banner also hides the DevTools URL in that case). Otherwise: clone `matt1398/claude-devtools` to `~/.local/share/claude-devtools`, pin to upstream's latest release tag, install pnpm@10 via `bun add -g` (upstream declares `packageManager: pnpm@10.x` and ships a pnpm-lock.yaml; bun mis-handles its workspace + optional deps), `pnpm install --frozen-lockfile && pnpm run standalone:build`, install + start `claude-devtools.service` on port 12002. Build runs under env hedges `MALLOC_ARENA_MAX=2 GOMEMLIMIT=2048MiB BUN_OPTIONS=--smol` to claw back a few hundred MB on borderline boxes. Build failures are non-fatal — any existing build keeps serving. Skipped under `--harden-only`. Bun remains the runtime (`bun run dist-standalone/index.cjs`). |
| 10 | Cleanup-only: if a prior setup installed `claude-run` (bun global) + `claude-history.service`, this phase stops/disables the service, removes the unit file, and runs `bun remove -g claude-run`. Runs in all modes; emits no output on a clean machine. |
| 11 | `apt autoremove` |

## Key Conventions

- **Idempotency**: every phase is safe to re-run.
- **install-if-missing-only**: if a tool is already present, don't reinstall and don't auto-update. The user manages tool versions externally (`uv tool upgrade litellm`, `claude update`, etc.).
- **Profile management**: use `update_profile_export` in `common.sh` for env-var changes — it writes to all `PROFILE_FILES` (just `~/.profile` by default). Never edit profile files with raw `sed`/`echo`. Never write provider secrets here — they belong in `~/.config/litellm/env`.
- **systemd template substitution**: service files in `linux/systemd/` use `__PLACEHOLDER__` tokens (`__LITELLM_BIN__`, `__APP_DIR__`, `__PORT__`, `__ENV_FILE__`, `__PATH__`). `deploy_user_systemd_service` in `common.sh` runs `sed` over them before `write_if_changed`.
- **Master-key strategy**: a `sk-<48 hex chars>` master key is auto-generated on first run via `openssl rand -hex 24`, written to `~/.config/litellm/env` (as `LITELLM_MASTER_KEY`) and to `~/.profile` (as `ANTHROPIC_AUTH_TOKEN` and `LITELLM_API_KEY`). All three names point at the same value. Re-runs preserve it via `read_profile_export`. Setting `ANTHROPIC_AUTH_TOKEN=sk-…` in `.env` overrides auto-generation.
- **Model naming**: the baseline `litellm-config.yaml` deliberately uses upstream provider-prefixed ids as Public Model Names (`azure/gpt-5.4`, `azure/gpt-5.4-mini`) — no `claude-*` alias layer. Claude Code asks for those ids directly via `ANTHROPIC_DEFAULT_{HAIKU,SONNET,OPUS}_MODEL` in `~/.profile`. Trade-offs accepted by this choice:
  - **Gateway-discovery picker auto-population is off** for the baseline entries. Claude Code's `/v1/models` discovery filters on names starting with `claude` or `anthropic` (see <https://code.claude.com/docs/en/llm-gateway>); upstream-prefixed names don't match. The entries still route fine — Claude Code just won't list them in `/model`. Typing `/model azure/gpt-5.4-mini` still works.
  - **Hardcoded Anthropic-shaped feature gates may turn off** (1M context, extended thinking, reasoning-effort control). Claude Code matches the model id against `claude-*`/`anthropic-*` patterns to decide which features to enable; upstream-prefixed ids don't match, so thinking/effort UI stays off and **no thinking block is ever sent** (the LiteLLM→Azure reasoning path never fires). Fixed by the `ANTHROPIC_DEFAULT_*_MODEL_SUPPORTED_CAPABILITIES` companions Phase 5a writes (`thinking,adaptive_thinking,interleaved_thinking,effort,xhigh_effort` — see Phase 5a for why `max` is excluded and haiku's list is shorter). Other gates (e.g. 1M context) have no published list — watch for unexpected token truncation.
  - **To get auto-discovery back for a UI-added model**, give it a `claude-*` or `anthropic-*` Public Model Name (e.g. `claude-gemini-3-pro` → `gemini/gemini-3.0-pro`, key `os.environ/GEMINI_API_KEY`). `CLAUDE_CODE_ENABLE_GATEWAY_MODEL_DISCOVERY=1` is left on in `~/.profile` so such an entry appears in `/model` automatically.
  - **Escape hatch for single-name picker entries**: set `ANTHROPIC_CUSTOM_MODEL_OPTION=<name>` (plus optional `…_NAME` / `…_DESCRIPTION` / `…_SUPPORTED_CAPABILITIES`) in `claude-managed-settings.json`'s `env:` block. Adds one custom picker entry regardless of name shape — useful for pinning the headline upstream id in the picker. Its `…_SUPPORTED_CAPABILITIES` companion is the lever to expose thinking/effort for that entry (same value shape as the `ANTHROPIC_DEFAULT_*` capabilities; arbitrary `/ui`-added models otherwise rely on `claude-*` naming for detection).
  - **No-Azure install path**: leaving `AZURE_OPENAI_API_KEY` / `AZURE_RESOURCE_ENDPOINT` blank in `.env` (or omitting `.env` entirely) is supported — setup still completes, the three `ANTHROPIC_DEFAULT_*_MODEL` exports in `~/.profile` are written empty (Phase 5a; see also the Azure section in `.env.example`), and a banner at the end of `setup.sh` tells the user to add a model via `http://127.0.0.1:4000/ui` and set the vars manually. Re-runs preserve any non-empty value already in `~/.profile`.
- **Managed-settings token substitution**: `linux/configs/claude-managed-settings.json` contains a `__REPO_DIR__` token (used by `hooks` paths). setup.sh Phase 8a substitutes via `sed` before sudo-installing to `/etc/claude-code/managed-settings.json`. `ANTHROPIC_BASE_URL`/`ANTHROPIC_AUTH_TOKEN` deliberately live in `~/.profile` only (not managed-settings) so the user can point at a remote router without root.
- **Telemetry**: the systemd unit pins `LITELLM_MODE=PRODUCTION`, `LITELLM_LOG=ERROR`, and passes `--telemetry False`. Per LiteLLM docs, self-hosted instances do not phone home, but we set the flag defensively. `LITELLM_LOCAL_MODEL_COST_MAP` is deliberately left unset so LiteLLM refreshes `model_prices_and_context_window.json` from `BerriAI/litellm@main` on every start — without this, the UI's "Add Model" provider dropdown is capped at the catalog frozen into the installed wheel (e.g. no `gpt-5.3` until you `uv tool upgrade litellm`). Tradeoff: a small GitHub fetch at boot.
- **Hardening boundaries**: this repo intentionally does **not** ship `bunfig.toml`/`npmrc`/`uv.toml`/`pip.conf`. Supply-chain hardening lives outside this repo.
- **Docker mode (`--docker`)**: an opt-in alternative runtime for **LiteLLM only** — Postgres stays a host system service and claude-devtools stays native. `install_docker_rootless` (common.sh) installs Docker CE and configures it **rootless** (`dockerd` as `systemd --user`); rootless is deliberate because adding the user to the `docker` group grants effective root via the daemon socket, which would undercut the managed-settings/sandbox/exit-2-hook posture. A pre-existing **rootful** daemon is coexisted-with, never disabled (the "user's call, not a silent side effect" stance documented in `install_docker_rootless`): when its socket is detected the setuptool gets `--force` so rootless installs alongside it on a separate per-user socket, and the litellm unit pins `DOCKER_HOST` to that rootless socket regardless of the default CLI context. The container is litellm only (`linux/configs/litellm-docker-compose.yml`, deployed to `~/.config/litellm/docker-compose.yml`), published **loopback-only** (`127.0.0.1:4000:4000`; the container listens on `0.0.0.0` so the mapping works, but it's never reachable off-host), mounting `./config.yaml` + `./env` (the same mode-600 env file the native unit uses). Image is a **pinned** `ghcr.io/berriai/litellm:vX.Y.Z` tag (currently `v1.86.0`) — bumped deliberately on a git edit + re-run, like the claude-devtools tag pin (a floating `latest`/`main-stable` would auto-roll-forward on the unit's `ExecStartPre` `docker compose pull`, against the no-auto-update convention). This pins LiteLLM's **plain per-release tag**, *not* the `-stable` channel: `-stable` tags are promoted later and **lag** (newest is `v1.83.14-stable`, *below* the native uv `>=1.84.0` floor — no `-stable` image exists for 1.84+), so a `-stable` pin can't match that floor. `v1.86.0` clears it, sits above the compromised 1.82.7/1.82.8, and carries the GPT-5.4 reasoning fixes (chat→Responses auto-route 1.83.0+, effort→`reasoning_effort` 1.83.1+). **When bumping, verify the exact tag exists on ghcr.io** (`curl -s "https://ghcr.io/token?scope=repository:berriai/litellm:pull"` → token → manifest HEAD) — a given version's `-stable` variant may not exist, which yields a `not found` pull error. **Host-DB connectivity uses the Unix socket, not TCP** — the most secure option and backend-independent (no reliance on rootless host-loopback / `host.docker.internal`): the host Postgres socket dir `/run/postgresql` is bind-mounted into the container and `DATABASE_URL` carries `?host=/run/postgresql`, so Postgres keeps its default loopback-only/socket binding and **nothing is exposed on any network interface**. The only host-side change is a scoped pg_hba line — `ensure_pg_socket_scram_rule` (common.sh, called from Phase 6) inserts `local litellm litellm scram-sha-256` above Debian's default `local all all peer` (idempotent; first-match-wins; reloads Postgres) because peer auth can't map the container's UID to the `litellm` role. We deliberately avoid `listen_addresses='*'` and never add a TCP `pg_hba` rule. Because the host DB is reused (not a fresh container volume), `/ui`-added models, keys, and spend history persist across a native↔docker switch. Footprint vs native: ~+2.4 GB disk (engine + ~2 GB image), ~+0.3–0.4 GB runtime RAM (container + rootless dockerd/containerd/slirp4netns); build-time RAM unaffected (nothing builds — claude-devtools' image isn't built on-host, sidestepping the vite OOM the ≥3.5 GB Phase 9 gate guards). The `litellm` user-service name is reused for both runtimes, so a re-run with/without `--docker` swaps `ExecStart` on the next `systemctl --user restart litellm`.
- **Sandbox control**: the entire `sandbox` block lives in the **user** file (`linux/configs/claude-settings.json` → `~/.claude/settings.json`), not managed-settings. This is deliberate so `/sandbox` actually works: per Claude Code's precedence, a sandbox **boolean** in managed settings (`enabled`, `failIfUnavailable`) is used verbatim and *ignores* any user/local value, so `/sandbox` (which writes `.claude/settings.local.json`) is inert whenever `sandbox.enabled` sits in managed — at any value, including `false` (confirmed behaviour; the picker also desyncs on custom sandbox blocks, anthropics/claude-code#47624). The template ships `enabled:true` (ON by default; `/sandbox` can toggle it off, local scope overriding the user default), `autoAllowBashIfSandboxed:true`, and the credential/network floor (`filesystem.denyRead`/`denyWrite`, `network.allowedDomains`/`deniedDomains`). `failIfUnavailable` is **omitted** (default `false`), so a box where bwrap can't run (e.g. a kernel with user namespaces disabled) warns + runs unsandboxed instead of refusing to start. `allowUnsandboxedCommands` is set to **`false`** (stricter than the `true` default — the `/sandbox` Overrides tab labels this **Strict sandbox mode**): the per-command `dangerouslyDisableSandbox` escape hatch is disabled (the parameter is completely ignored), so a sandboxed command must run sandboxed or not at all (barring an `excludedCommands` entry). Deployed fresh-install-only (Phase 8c guard): full/harden ship `enabled:true`, `--router-only` ships the **same block with `enabled:false`** (`jq '.sandbox.enabled = false'` — **not** stripped, so the floor stays pre-configured; flip `enabled:true` to turn it on and the statusline detects it). On an **existing** settings.json with no sandbox block, setup never rewrites the CC-owned file — the end-of-setup banner points at `linux/scripts/add-sandbox-block.sh` (idempotent `//=` add-if-missing merge; `--disabled` under router-only) so the user can opt in without losing their keys. (The `/sandbox` picker is **not** a reliable persistence path — it frequently no-ops + desyncs, anthropics/claude-code#47624 — and its toggle is invisible to the statusline (runtime-only, #30772); persisting `enabled` in user settings is the detectable mechanism.) The bwrap runtime is still installed in every mode (Phase 2). **Security note**: credential protection does **not** depend on the sandbox — the managed `permissions.deny[]` `Read(...)` rules (root-enforced, non-removable) cover the credential paths regardless. The sandbox `filesystem` floor is user-editable defense-in-depth that only applies while the sandbox is on; because the user can now disable the sandbox entirely, it is a strong **default** but not an **enforced** boundary. **Linux glob caveat**: the six leading-`**/` `Read(...)` globs in the managed `permissions.deny[]` (`**/.env`, `**/.env.*`, `**/*.pem`, `**/*.key`, `**/id_rsa`, `**/id_ed25519`) trigger a Linux startup warning ("Glob patterns in sandbox permission rules are not fully supported") because bubblewrap can't expand a wildcard-directory prefix into a fixed path to mask — fixed-prefix globs (`~/.ssh/**`) and the literal `filesystem.denyRead` paths translate fine and aren't flagged. It is **not** a misconfiguration: the **permission layer still enforces these globs** (Claude's Read/Edit tool + the bash readers it recognises — `cat`/`head`/`tail`/`sed`), so credential protection holds; only bubblewrap's FS layer drops them. Residual gap = an arbitrary subprocess *inside* the sandbox (`python`, `node -e`, a script opening the file itself) reading a project-relative `.env`/`*.pem`/`*.key`, mitigated by the literal credential-dir `denyRead` floor + the `network.allowedDomains`/managed `curl`/`wget`/`nc` denies that throttle exfiltration. Known won't-fix (anthropics/claude-code#32226, #56698); no suppression flag. The globs are **kept deliberately** — deleting them is the only way to silence the warning and would strip the load-bearing permission-layer protection (same principle as `feedback-explicit-deny-rules`). See *Troubleshooting* for the user-facing note.
- **Action-aware permission layer (`nah`)**: Phase 8e installs the `nah` Claude Code plugin (`manuelschipper/nah`, unpinned) as an additive PreToolUse guard. Stacks alongside `claude-managed-settings.json` denies. Per Anthropic's docs (code.claude.com/docs/en/permissions, hooks-reference), the evaluation order is: **PreToolUse hooks fire first**, then permission rules in order `deny → ask → allow`. A hook returning `"deny"` (or exit 2) blocks and **also overrides `allow[]`** rules; a hook returning `"allow"` cannot override `deny[]` or `ask[]` — those still fire. So both layers act as independent gates: the regex denies catch everything that matches their glob shape, and `nah` catches wrapper-evasion cases the regexes miss (`bash -c`, `python -c`, `xargs rm`, `find -delete`, `git push -f` short form, sudo-wrapped commands; content-scan: inline `python3 -c`/`node -e` code). nah's defaults (`src/nah/data/policies.json`) are looser than this repo's blanket denies on `pip install`, `curl *`, `git push --force*` — and since a hook `"allow"` cannot override `deny[]`, the regex denies remain load-bearing. Do **not** remove them in favor of nah (auto-memory entry `feedback-explicit-deny-rules` formalises the same principle). The repo also keeps its own `pretooluse-rm-guard.sh` + `pretooluse-push-guard.sh` exit-2 hooks (deployed via managed-settings, root-enforced) alongside nah — not redundant: under `--dangerously-skip-permissions` the `permissions.deny[]` rm rules are skipped and nah's `filesystem_delete: context` likely allows non-sensitive deletes, so the repo-owned exit-2 rm-guard adds a narrow deterministic floor there: it hard-blocks **direct** recursive-force `rm` (`rm -rf`/`-Rf`/`--recursive --force`, any flag order/case) even in bypass mode (PreToolUse hooks still fire there). It does **not** catch wrapper-evasion forms (`sudo rm -rf`, `bash -c '… rm -rf'`, `xargs rm`, `find -delete`) — those rely on nah's action/content scan (whose `filesystem_delete: context` may still allow non-sensitive deletes) and ultimately the bubblewrap sandbox. It is also a fallback if nah is uninstalled/disabled/regresses. Known caveats: (1) **Under `--dangerously-skip-permissions` / `bypassPermissions`**: PreToolUse hooks still fire per Anthropic docs and community confirmation — the bypass flag skips the prompt + the deny/ask/allow rule chain, but hooks run *before* the prompt and remain active. So nah's `"deny"` / exit-2 decisions still block tool calls in this mode, making nah the **only** active policy layer (the `permissions.deny[]` regex rules are bypassed too). Hook `"ask"` decisions likely fall through to allow since prompts are off; only hook `"deny"` actually blocks. Known interaction bugs to watch: anthropics/claude-code#37745 (bypass can reset mid-session when hooks are configured), #47810 (both layers stop firing after background tasks complete), #40241 (subagents spawned via the Agent tool don't inherit bypass — nah hooks may not run there). Claude Code refuses to start under `--dangerously-skip-permissions` when running as root/sudo on Linux/macOS — a hard constraint. Only the bubblewrap sandbox (`sandbox.enabled`) is a kernel-level boundary that survives subagent escapes and hook regressions — but note it now lives in **user** settings (see *Sandbox control*) and is ON by default yet user-disableable via `/sandbox`, so it is no longer an *enforced* boundary; treat the managed `permissions.deny[]` + the exit-2 hooks as the load-bearing layer; (2) plugin form has no `.nah.yaml` tuning, only upstream defaults; (3) marketplace ref uses the `@claude-marketplace` branch (where upstream's `marketplace.json` lives — the default branch doesn't have it) and is otherwise unpinned; resolves on first `setup.sh` run only (re-runs skip on "already added"), so a hostile upstream update lands on fresh installs only — switch to `anthropics/claude-plugins-community` once `nah` lands there (Anthropic owns commit-SHA pin); (4) user-uninstall via `claude plugin uninstall nah --scope user` is re-installed (and re-enabled) on next `setup.sh`; but a user-**disable** via `claude plugin disable nah --scope user` is now **respected** — Phase 8e leaves an installed-but-disabled plugin disabled (install enables by default + the absent branch re-asserts enable, so a disabled state can only be a deliberate opt-out). To opt out: `disable` it (survives re-runs), or `uninstall` it and comment Phase 8e out to prevent reinstall.

- `~/.local/bin/litellm` — LiteLLM CLI (uv-installed)
- `~/.local/share/uv/tools/litellm/` — uv-managed virtualenv for LiteLLM and its deps
- `~/.config/litellm/{config.yaml, env}` — LiteLLM runtime state (env file holds master key, auto-discovered provider secrets, `DATABASE_URL`, `STORE_MODEL_IN_DB=True`, `LITELLM_DB_PASSWORD`). Under `--docker`, also `~/.config/litellm/docker-compose.yml` (rendered from `linux/configs/litellm-docker-compose.yml`; mounts `config.yaml` + `env`)

- `~/.local/share/claude-devtools/` — claude-devtools clone + build output (`dist-standalone/index.cjs`); `.dt-installed-tag` stamps the active release
- `~/.config/systemd/user/{litellm.service, claude-devtools.service, docker.service}` — systemd user units (`docker.service` only under `--docker`; rootless daemon set up by the setuptool. The `litellm` unit is rendered from `litellm.service` natively or `litellm-docker.service` under `--docker`)
- `/etc/claude-code/managed-settings.json` — system-level Claude Code policy (full + harden-only; skipped under `--router-only`; root-owned)
- `~/.claude/{settings.json, statusline.sh, projects/}` — user-level Claude Code state
- `~/.profile` — gateway URL, master key (as `ANTHROPIC_AUTH_TOKEN`), telemetry opt-outs (sourced by bash + zsh login shells)

## Troubleshooting

If LiteLLM fails to start, the service runs at `LITELLM_LOG=ERROR` which hides Prisma's "Run `prisma migrate resolve …`" recovery hints. To see them, edit `~/.config/systemd/user/litellm.service`, change `LITELLM_LOG=ERROR` to `LITELLM_LOG=INFO`, `systemctl --user daemon-reload && systemctl --user restart litellm`, and read `journalctl --user -u litellm -n 200`. Revert when done — INFO is very chatty in steady state.

**Statusbar shows `statusline skipped · restart to fix` and no trust prompt appears.** Caused by `IS_DEMO=1` leaking into the shell — Claude Code treats it as a "demo session" marker that silently suppresses the workspace-trust dialog without granting trust, so `statusLine` + hooks stay gated off and a restart doesn't help (anthropics/claude-code#37780, same flaw pattern as the fixed #10409). Phase 5a now strips it via `remove_profile_export "IS_DEMO"` (deletes the export from `~/.profile` and `unset`s it from the current shell). On a machine that ran an older setup, also check anywhere else it might be exported (`~/.bashrc`, `~/.zshrc`, container entrypoints, dotfiles), then relaunch `claude` and accept the trust prompt. As defense-in-depth Phase 8c only deploys `~/.claude/settings.json` on a fresh install — once Claude Code owns the file, re-running setup leaves it alone.

**Startup warning: "Glob patterns in sandbox permission rules are not fully supported on Linux."** Expected, and harmless to enforcement — not a misconfiguration. The six leading-`**/` `Read(...)` rules in `claude-managed-settings.json` (`**/.env`, `**/.env.*`, `**/*.pem`, `**/*.key`, `**/id_rsa`, `**/id_ed25519`) still block Claude's Read/Edit tool at the permission layer; only bubblewrap's filesystem layer can't apply them on Linux (no fixed path to mask). The residual gap — an arbitrary subprocess inside the sandbox reading a project-relative secret — is covered by the literal credential-dir `denyRead` floor plus the network-egress denies. There is **no flag to silence** the warning; removing the globs is the only way and would weaken the load-bearing protection, so we keep them (see *Sandbox control*). Refs: anthropics/claude-code#32226, #56698.
